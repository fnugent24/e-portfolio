Team activity - What is a Secure Programming Language:

What factors determine whether a programming language is secure or not?

According to Cifuentes and Bierman (2019), the 3 most common software vulnerabilities they found from the National Vulnerability Database (NVD) data were buffer errors (including buffer overloads), injection errors (including cross-site scripting (XSS), SQL injection, code injection), and information leak errors (including information leaked through log files, debugging, test code). According to them, a truly secure programming language is ‘one that has first-class support for all three categories’. In other words, a secure programming language is one that has no cases of buffer errors, injection errors and/or information leak errors. Therefore, these categories are suggested to be the factors determining whether a programming language is secure or not.

While Pillai (2017) suggests that these three factors contribute to determining whether a programming language is secure, he also notes other factors that play a role, including: cryptography issues, authentication issues, and race conditions.

Could Python be classed as a secure language?

Cifuentes and Bierman (2019) argue that Python cannot be classed as a secure language due to the implication that it does not hold to what they define as a secure language (i.e., it is vulnerable to buffer errors, injection errors and information leak errors).

Pillai (2017) argues that the classification of Python as a secure language ‘depends on the application code and the dependent module code used, and how they are able to deal with or mask the…errors.’ For example, using the input value in Python 2.x can expose sensitive data in the code, including the passwords of some users in the system. Pillai notes that ‘even if there were no sensitive data in the program, a hacker using this approach can reveal valuable information about the program such as variable names, function names, packages used, and so on.’ Yet, this can be fixed by replacing the input value with “raw_input”, however, the developers still need to perform proper type conversion to maintain security levels.

Pillai also importantly notes that many security vulnerabilities that occur/occurred in the older (but still widely used) Python 2.x versions have been fixed in the more recent versions (3.x) of Python, so this is important to keep in mind when evaluating the nature of Python as a secure language.

Would Python be a better language to create operating systems than C? Why/why not?

Python would not be a better language since it is a high-level and interpreted language. C, on the other hand, is a low-level language, which makes it significantly easier to create an operating system with.

Buffer Overflow in Python:

In the Buffer Overflow in Python file in Codio, the code in the Overflow.py file reads: 


                        buffer=[None]*10
                        for i in range (0,11):
                            buffer[i]=7
                        print(buffer)

When I run the file it reads: 


                        Traceback (most recent call last):
                          File "Overflow.py", line 3, in 
                            buffer[i]=7
                        IndexError: list assignment index out of range

When I install pylint (i.e. writing in the command “pip3 install pylint”) and run the same file, it reads: 


                        ************* Module Overflow
                        Overflow.py:4:0: C0303: Trailing whitespace (trailing-whitespace)
                        Overflow.py:5:0: C0304: Final newline missing (missing-final-newline)
                        Overflow.py:1:0: C0103: Module name "Overflow" doesn't conform to snake_case naming style (invalid-name)
                        Overflow.py:1:0: C0114: Missing module docstring (missing-module-docstring)
                        
                        -----------------------------------
                        Your code has been rated at 0.00/10

Explanation:

The first sentence (Overflow.py:4:0: C0303) basically means that line 4 at column 0 violates the convention C0303.

The second sentence (Overflow.py:5:0: C0304) basically means that line 5 at column 0 violates the convention C0304.

The fourth sentence (Overflow.py:1:0: C0114) basically means that line 1 at column 0 violates the convention C0114.

Also, the information showing the message of the error is important (e.g. “missing-final-newline”)

We can use the help message on the command line (e.g. pylint --help-msg=missing-final-newline) and this will tell us what the error is: 


                        pylint --help-msg=trailing-whitespace
                        :trailing-whitespace (C0303): *Trailing whitespace*
                          Used when there is whitespace between the end of a line and the newline. 
                        This message belongs to the format checker.

Or using “pylint --help-msg=missing-module-docstring” will tell us: 


                          pylint --help-msg=missing-module-docstring
			:missing-module-docstring (C0114): *Missing module docstring*
			  Used when a module has no docstring.Empty modules do not require a docstring.
			  This message belongs to the basic checker.

So this error is telling us that we are missing a docstring at the beginning of the code. If we did add a docstring (e.g. “““This script is telling us…”””) to the code and run the file again, it says: 


			Your code has been rated at 2.50/10 (previous run: 0.00/10, +2.50)

So now, we have eliminated that error and the score of the code has gone up!
